Every cluster — official or community — uses the same domain naming pattern. The cluster slug differentiates them; root domains (no slug) reconcile against active platform-official clusters when Navigator runs without a tenant context.

## Domain Pattern

| Domain                      | Level   | Purpose                                      |
| --------------------------- | ------- | -------------------------------------------- |
| `foghorn.{slug}.{base}`     | Cluster | Points to this cluster's Foghorn instance(s) |
| `livepeer.{slug}.{base}`    | Cluster | Points to this cluster's Livepeer gateway(s) |
| `edge.{slug}.{base}`        | Cluster | Points to this cluster's edge pool           |
| `edge-ingest.{slug}.{base}` | Cluster | Points to this cluster's ingest nodes        |
| `edge-egress.{slug}.{base}` | Cluster | Points to this cluster's edge nodes          |

Media domains are scoped to a media cluster slug: `edge`, edge capability services, `telemetry`, `chandler`, `foghorn`, and `livepeer`.

## Protocol Routing

HTTP-based protocols (HLS, DASH, WHEP, MP4) go through Foghorn, which issues 307 redirects to the nearest edge node. Non-HTTP protocols (SRT, RTMP, RTSP, DTSC) connect directly to edge nodes — they can't follow HTTP redirects.

| Protocol                   | Domain                      | Transport                 |
| -------------------------- | --------------------------- | ------------------------- |
| HLS, DASH, WHEP, MP4, WebM | `foghorn.{slug}.{base}`     | HTTP 307 redirect to edge |
| Livepeer gateway API       | `livepeer.{slug}.{base}`    | HTTP status/API endpoint  |
| SRT (ingest)               | `edge-ingest.{slug}.{base}` | Direct UDP                |
| RTMP (ingest)              | `edge-ingest.{slug}.{base}` | Direct TCP                |
| SRT (playback)             | `edge-egress.{slug}.{base}` | Direct UDP                |
| RTSP, DTSC                 | `edge-egress.{slug}.{base}` | Direct TCP                |

## DNS Configuration

### TTL

Navigator defaults both A-record and Cloudflare load-balancer TTLs to 60 seconds. Shorter TTLs increase DNS query volume without meaningful failover improvement.

### Health Checks

- **Endpoint**: `GET` to the service health path, usually `/health`; service-specific paths come from the service registry
- **Port**: the service's default HTTP health port; edge pools use `18008`
- **Interval**: 60 seconds by default (`NAVIGATOR_CF_MONITOR_INTERVAL`)
- **Timeout**: 5 seconds by default (`NAVIGATOR_CF_MONITOR_TIMEOUT`)
- **Failure threshold**: 2 consecutive failures by default (`NAVIGATOR_CF_MONITOR_RETRIES`)
- **Expected response**: HTTP `200`

### TLS Certificates

Use wildcard certificates issued via DNS-01 challenge (Let's Encrypt or equivalent):

- `*.{base}` — covers root-level domains
- `*.{slug}.{base}` — covers per-cluster domains

<Aside type="tip">
  DNS-01 validation works regardless of whether HTTP is reachable, making it suitable for clusters
  behind firewalls or in private networks.
</Aside>

## GeoDNS Strategy

Navigator currently implements DNS through Cloudflare:

1. Quartermaster returns DNS-eligible nodes with non-empty `external_ip`.
2. A service with one healthy IP gets an A record.
3. A root/global service with two or more healthy IPs gets a Cloudflare load balancer with one pool per Quartermaster cluster.
4. Pool `latitude` / `longitude` values are set from the centroid of healthy node coordinates, and the load balancer uses `steering_policy: proximity` when at least two pools have coordinates.
5. Cluster-scoped domains use one pool for that cluster and balance across healthy origins inside the pool.
6. Cloudflare health monitors suppress externally unreachable origins between Navigator reconciliations; Quartermaster remains the desired-state source.

<Aside type="note">
  GeoDNS is a stopgap until anycast DNS with a custom backend resolver is justified — typically at
  10+ PoPs where sub-30-second failover matters.
</Aside>

## How the Frontend Uses This

The GraphQL `streamingConfig` query returns cluster-specific domains for the authenticated tenant. Chartroom uses these to build protocol-specific URLs, falling back to `VITE_STREAMING_*` env vars when the query returns null (dev environment, unauthenticated users, or Quartermaster unavailable).

The player and studio SDKs are unaffected — they use `resolveViewerEndpoint` / `resolveIngestEndpoint` for runtime node resolution and never read these domains.
